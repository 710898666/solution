# 框图设计工具 — 产品上下文

> 本 Skill 为 Claude 提供框图设计工具的产品定位、用户画像、工具链关系和核心设计原则，确保所有讨论建立在一致的认知基础上。

---

## 1. 产品定位：一句话

**框图设计工具是系统级硬件架构设计工具，位于传统 EDA 工具（Altium、Cadence、Allegro）的上游，让架构师通过 Block Diagram、接口契约和结构化数据表达设计意图，并将其作为可机器消费的 Design Intent Package 向下游传递。**

### 1.1 本工具「不是」什么

| 本工具不是 | 区别 |
|---|---|
| 原理图工具 | 不处理器件级 Symbol / Footprint / 布局布线 |
| PCB 设计工具 | 不做 Layout、DRC、信号完整性仿真 |
| 项目管理工具 | 不是 Jira / PLM，虽涉及协作但核心是设计数据 |
| Visio / Draw.io | 不是自由画布，Block Diagram 背后有严格数据模型 |
| BOM 管理工具 | 不是主数据源，但可输出/关联 BOM 信息 |

### 1.2 本工具「是」什么

本工具的本质是一个**结构化的设计意图表达与传递平台**：

- **表达**：让架构师把脑中的系统架构变成结构化、可校验的数据，而非仅停留在 PPT 和白板
- **传递**：通过 Design Intent Package（DIP）单向输出给下游 EDA 工具，可选反向校验（下游实现结果与上游意图是否一致），但不做实时双向同步
- **协作**：多角色（架构师、硬件工程师、SE）基于同一套结构化数据工作，消除信息断层

---

## 2. 核心用户角色

### 2.1 系统架构师（主要用户）

- **场景**：定义系统级 Block Diagram，划分功能模块边界，指定模块间接口契约（协议、信号、Pin 分配），管理多方案变体（高/标/低配）
- **痛点**：设计意图只能用 PPT/Visio 表达，无法结构化、无法校验、无法被下游工具直接消费
- **工具价值**：将架构意图从"图片"升级为"数据"

### 2.2 硬件工程师（协作用户）

- **场景**：接收架构师的设计意图，在 EDA 工具中落地实现，需要明确的接口定义和约束
- **痛点**：架构师的意图靠口头/文档传递，信息丢失和误解频发
- **工具价值**：获得结构化的 DIP 作为设计输入，减少返工

### 2.3 项目经理 / SE（管理用户）

- **场景**：追踪方案变体状态，审批设计方案，进行跨模块/跨团队的接口协调
- **痛点**：缺乏全局视图，方案比选靠手工对比
- **工具价值**：多变体管理 + 全局架构视图

## 3. 工具链定位

```
框图设计工具
  ├── 输入源：需求文档、系统规格、接口标准
  ├── 核心产出：Design Intent Package（DIP）
  │     ├── 结构化 Block Diagram（含层级关系）
  │     ├── 接口契约（Protocol + Signal + Pin Mapping）
  │     ├── 多变体方案定义（SKU variants）
  │     └── 设计规则 & 约束
  ├── 下游消费者
  │     ├── 原理图工具（Altium / Cadence / Allegro）← 单向导出
  │     ├── 机械工具（Creo 等）← 边界/尺寸约束
  │     └── 文档系统 ← ICD / 接口规格书
  └── 可选反向校验
        └── 下游实现结果 → 框图设计工具校验是否符合原始意图
```

### 3.1 集成策略核心原则

- **单向导出为主**：框图设计工具 → EDA 工具，通过 DIP 传递
- **可选反向校验**：EDA 工具的实现结果可回传框图设计工具做一致性校验
- **不做实时双向同步**：这是一个明确的架构决策，避免耦合复杂度爆炸

---

## 4. 核心痛点与优先级

按优先级从高到低排列（此排序为产品战略共识）：

1. **架构意图无法结构化表达和传递** — 最核心的存在理由
2. **跨角色协作信息断层** — 架构师 ↔ 硬件工程师之间的信息丢失
3. **方案比选/多变体管理缺失** — 高/标/低配方案无法结构化对比
4. **设计规则校验前移到架构阶段** — 把部分 DRC 前置，减少下游返工

## 5. 核心概念速查

> 详细数据模型见 references/data-model.md（如已创建）

| 概念 | 定义 | 注意事项 |
|---|---|---|
| **Block** | 功能模块的抽象表示，可嵌套 | CBB 和器件转化的 Block 默认泛型矩形 |
| **Port** | Block 上的接口端口，承载一组 Signal | Port ≠ Signal，必须显式区分 |
| **Signal** | Port 内的具体信号线，**用户日常操作的核心对象** | 一个 Port 可包含多个 Signal；一个 Signal 可包含多个 Pin（多:1 聚合） |
| **Pin** | Signal 的附属数据，物理引脚的映射 | 连接物理世界的最后一层；用户通常不直接操作 Pin，而是通过 Signal 间接管理 |
| **Board** | 项目级物理实体（PCB 板） | 不是视觉分组框，是物理概念 |
| **CBB** | Common Building Block，可复用模块 | 默认泛型矩形 + 非阻塞形状建议 |
| **DIP** | Design Intent Package，设计意图包 | 框图设计工具的核心输出物 |
| **ICD** | Interface Control Document | 可从框图设计工具数据自动/半自动生成 |
| **Protocol** | 标准接口协议（如 I2C、SPI、USB 等） | 协议字典支持 19+ 标准接口 |
| **Variant/SKU** | 方案变体（如高配/标配/低配） | 多变体共享基础架构，差异化配置 |

## 5.1 数据模型层级（确认演进方向）

当前代码中的模型为 `Block → Port → Pin`（Pin 直挂 Port）。确认的演进方向为四层模型：

```
Block → Port → Signal → Pin
```

具体变更：`Port.business.pins` → `Port.business.signals`（每个 Signal 内含 `pins: Pin[]`）；`Block.business.freePins` → `Block.business.freeSignals`。此演进是 P2 原则（Port 与 Signal 显式区分）的进一步落地。

### 5.2 器件 Block 的数据流链路

器件导入后的原始数据只有 Pin。完整处理链路为两次汇聚：

```
器件 Pin 列表
  → 汇聚引擎自动聚合（Pin → Signal，基于字典集）
  → Signal 列表
  → 汇聚引擎自动聚合（Signal → Port，基于字典集）
  → Port 结构
```

两次汇聚后均支持用户持续编辑（调整 Pin↔Signal 归属、调整 Signal↔Port 归属）。

### 5.3 孤儿 Pin 处理规则

汇聚引擎未识别的 Pin **自动 1:1 升格为同名 Signal**，不阻塞后续流程。用户可在后续编辑中将孤儿 Signal 合并到其他 Signal 或重命名。

### 5.4 三类 Block 的编辑模式

| Block 类型 | 编辑范围 | 原因 |
|---|---|---|
| 器件 Component | Pin↔Signal + Signal↔Port（两层） | 原始数据只有 Pin，需经两次汇聚 + 用户编辑 |
| CBB | Signal↔Port（为主） | CBB 库自带 Signal 甚至 Port 结构，少量调整 |
| 通用 General | Signal↔Port（从零建） | 无预设 Pin 数据，用户直接创建 Port 和 Signal |

## 6. 核心设计原则

这些原则在所有产品讨论中具有约束力：

### P1: 结构一致性优先
> 保持数据结构的一致性，让个别功能去适配结构，绝不为单个功能的便利而打破结构。

### P2: Port 与 Signal 显式区分
> Port 和 Signal 是两个独立层级，任何 UI 和数据模型设计都不得将二者混淆或隐式合并。

### P3: 单向导出 + 可选反向校验
> 与下游 EDA 工具的集成策略是单向导出 DIP，不做实时双向同步。反向校验是可选能力。

### P4: CBB 默认泛型矩形
> CBB 和器件转化的 Block 在画布上默认显示为泛型矩形，可提供非阻塞的形状建议，但不强制。

### P5: 页面级 JSON + 同步引擎
> 每个页面存储为独立 JSON 文件，跨页 Block 引用通过 sharedBlockId 字段 + 同步引擎实现，不采用单一数据源架构。

### P6: 标注元素页面级
> Annotation 元素（标注、备注等）仅在页面级别存在，不跨页传播。

### P7: 组件面板双区设计
> 组件面板分为固定模板区（常用 Block 模板）和搜索驱动的库区（CBB 库/器件库）。

## 7. 与 HScope 的关系

HScope 是内部的 PCB / 单板设计工具，框图设计工具与其关系：

- **框图设计工具在上游**：框图设计工具产出的 DIP 可作为 HScope 的设计输入
- **HScope 的核心问题**（已识别）：时序错配、数据源错误、模块缺乏共享数据骨干、缺少 scope-filtering 产品框架
- **演进方向**：HScope 应锚定跨角色协调场景，其 2D 图表模块应作为结构化数据生产者而非静态图片工具
